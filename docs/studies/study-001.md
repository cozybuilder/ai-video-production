## 1. 핵심 요약

Study 001은 Claude와 Higgsfield MCP를 연결해 AI 영상 제작 워크플로우를 자동화·시스템화하는 흐름을 다룬다. 핵심은 “영상 프롬프트 하나를 잘 쓰는 법”이 아니라, Claude를 중심 작업공간으로 두고 Higgsfield 같은 영상 도구를 연결해 **아이디어 → 에셋 → 장면 → 영상**으로 이어지는 생산 파이프라인을 만드는 것이다.

이 Study에서 배워야 할 핵심은 다음이다.

1. Claude를 단순 텍스트 챗봇이 아니라 제작 오케스트레이터로 사용한다.
2. MCP/연동 도구는 반복 작업을 줄이고 워크플로우를 연결하는 데 쓴다.
3. 영상 생성은 단발 프롬프트가 아니라 장면 단위 시스템으로 관리한다.
4. 사람은 Claude에게 명확한 목표·제약·출력 형식을 주고, 결과를 반복 개선한다.
5. AI 영상 제작은 “좋은 프롬프트”보다 “좋은 작업 구조”가 먼저다.

## 5. SRT 분석 — Workflow

### Step 1. 목표를 Claude에 명확히 입력한다

사람은 Claude에게 단순히 “멋진 영상 만들어줘”라고 하지 않는다. 어떤 영상인지, 어떤 목적의 영상인지, 어떤 도구로 보낼 것인지 목표를 먼저 지정한다.

패턴:

```text
목표 영상 유형 → 사용할 도구 → 출력 형식 → 장면 구조
```

### Step 2. Claude가 제작 구조를 만든다

Claude는 최종 영상 생성기가 아니라 중간 제작 문서를 만드는 역할을 한다.

- 아이디어 정리
- 장면 리스트 생성
- 영상 프롬프트 구조화
- 도구별 입력 형식 변환
- 수정 지시 정리

### Step 3. Higgsfield/Fable 같은 영상 도구에 연결한다

Claude가 만든 구조를 영상 도구에 넣어 장면을 생성한다. 이때 사람의 역할은 결과를 보고 깨지는 부분을 찾아 다시 Claude에게 수정시키는 것이다.

패턴:

```text
Claude 구조화 → 영상 도구 생성 → 결과 검토 → Claude 수정 → 재생성
```

### Step 4. 결과를 반복 개선한다

한 번에 완성하지 않고, 장면별로 문제를 좁혀 고친다.

- 인물 일관성
- 카메라 동작
- 장면 연결
- 제품/오브젝트 유지
- 영상 톤

## 6. 사람이 AI에게 입력한 핵심 지시 패턴

아래는 Claude/AI가 생성한 최종 결과물이 아니라, 사람이 Claude/AI를 어떻게 조종하는지에 대한 패턴이다.

### 6.1 역할 부여형 지시

```text
너는 AI 영상 제작 디렉터다.
너는 프롬프트 엔지니어다.
너는 광고 영상 기획자다.
```

목적: Claude가 단순 답변자가 아니라 제작 전문가 관점으로 사고하게 만든다.

### 6.2 목표 정의형 지시

```text
이 영상을 수익화 가능한 AI 광고/콘텐츠로 설계해라.
```

목적: 영상의 목적을 미리 고정한다.

### 6.3 도구 연결형 지시

```text
Claude가 만든 결과를 Higgsfield/Fable에 넣을 수 있는 형태로 정리해라.
```

목적: Claude 출력물이 바로 다음 도구의 입력이 되게 한다.

### 6.4 출력 형식 지정

```text
장면별로 나눠라.
shot list 형태로 정리해라.
prompt ID를 붙여라.
```

목적: 긴 문장을 관리 가능한 제작 단위로 쪼갠다.

### 6.5 반복 수정형 지시

```text
이 결과에서 문제가 되는 장면만 수정해라.
전체를 다시 쓰지 말고 특정 cut만 바꿔라.
```

목적: 생산 속도를 유지하면서 품질을 높인다.

### 6.6 플랫폼 최적화 지시

```text
Instagram Reels / YouTube Shorts / TikTok용 세로 영상으로 맞춰라.
```

목적: 영상 생성 전부터 최종 업로드 환경을 반영한다.

### 6.7 에셋 고정 지시

```text
같은 캐릭터와 같은 제품이 유지되도록 reference를 고정해라.
```

목적: AI 영상의 일관성 붕괴를 줄인다.

### 6.8 카메라 언어 지시

```text
close-up, medium shot, push-in, tracking shot처럼 카메라를 명확히 써라.
```

목적: “시네마틱하게” 같은 추상어를 실제 영상 언어로 바꾼다.

### 6.9 실패 원인 분석 지시

```text
왜 결과가 어색한지 분석하고, 프롬프트를 어떻게 고쳐야 하는지 말해라.
```

목적: 실패를 다음 프롬프트 개선에 연결한다.

### 6.10 시스템화 지시

```text
이 과정을 다음 영상에도 재사용할 수 있는 템플릿으로 만들어라.
```

목적: 단발 제작이 아니라 반복 가능한 파이프라인을 만든다.

### 6.11 수익화 관점 지시

```text
이 워크플로우를 광고, 숏폼, 클라이언트 작업에 적용할 수 있게 정리해라.
```

목적: 영상 제작을 취미가 아니라 상품화 가능한 서비스로 본다.

### 6.12 문서화 지시

```text
제작 단계, 입력값, 출력값, 실패 수정법을 문서로 남겨라.
```

목적: 같은 실수를 반복하지 않고 팀/도구 간 인수인계가 가능하게 한다.

## 7. 영상에서 배울 운영 원칙

### 7.1 Claude는 중심 제작 문서다

Claude는 프롬프트 몇 줄을 만들어주는 도구가 아니다. 전체 제작 흐름을 잡고, 각 장면을 관리하며, 수정 이력을 이어가는 중심 문서 역할을 해야 한다.

### 7.2 MCP/도구 연결은 반복 작업을 줄이는 데 쓴다

도구 연결의 목적은 멋있어 보이는 자동화가 아니라, 사람이 매번 복사·붙여넣기·형식 변환하는 시간을 줄이는 것이다.

### 7.3 영상은 장면 단위로 관리해야 한다

하나의 긴 프롬프트보다 장면 ID, 컷, 목적, 카메라, 에셋, 수정 지시가 분리된 구조가 낫다.

### 7.4 좋은 결과는 생성보다 수정 시스템에서 나온다

첫 결과가 완벽할 가능성은 낮다. 실패를 분석하고 특정 장면만 수정하는 구조가 핵심이다.

### 7.5 수익화는 영상 퀄리티보다 반복 생산성에서 시작된다

하나를 잘 만드는 것도 중요하지만, 같은 구조로 계속 만들 수 있어야 비즈니스가 된다.

## 8. HARIN 프로젝트 적용안

### 8.1 HARIN 제작의 중심은 클챗 문서다

HARIN 영상 제작은 Higgsfield에서 바로 시작하지 않는다.

```text
클챗 기획 문서
↓
HARIN 에셋 정의
↓
장면 리스트
↓
Higgsfield 프롬프트
↓
결과 검수
↓
수정 지시
```

### 8.2 HARIN 영상 템플릿

```text
Project:
Platform:
Duration:
Goal:
Product:
HARIN locked assets:
Scene list:
Prompt IDs:
Risk:
Correction prompt:
Caption:
Hashtags:
```

### 8.3 HARIN 적용 규칙

1. 영상 시작 전 HARIN reference를 고정한다.
2. 제품/장소/의상도 에셋으로 분리한다.
3. 클챗은 shot list editor로 사용한다.
4. Higgsfield는 장면 생성기로 사용한다.
5. 결과가 깨지면 전체 재작성 금지, 특정 prompt ID만 수정한다.
6. 최종 업로드 문구까지 같은 프로젝트 문서에 남긴다.

### 8.4 HARIN용 실행 프롬프트

```text
You are the AI video production director for SUH HARIN.

Build a vertical short-form ad workflow.
Do not write one long prompt.
Create:
1. asset list
2. 5-scene shot list
3. Higgsfield-ready prompt per scene
4. likely failure points
5. correction prompts
6. upload caption and hashtags
```
